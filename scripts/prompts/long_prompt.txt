A compiler turns a program written in one language into a program written in another, and almost every line of code you have ever run has been through one. The input is usually a high level language that humans find easy to write and reason about, things like Python or Rust or C, and the output is something a machine can execute directly, either machine code targeting a specific processor or an intermediate form like bytecode that runs on a virtual machine. The path from source text to executable is rarely a single jump. It is broken into stages, each of which transforms the program into a slightly different shape, and each of which has its own set of decisions and trade offs.

The first stage is lexical analysis, sometimes called scanning. The compiler reads the raw source text one character at a time and groups those characters into tokens. A token is the smallest meaningful unit of the language, things like a keyword, a number, an identifier, an operator, or a punctuation mark. Whitespace and comments are typically discarded at this stage, because they have no effect on the meaning of the program. The output of the lexer is a stream of tokens, each tagged with what kind of thing it is and where in the source file it came from. Source location is important because later stages need to be able to report errors back to the programmer in terms of the original file.

The second stage is parsing. The parser takes the stream of tokens and assembles them into a tree structure called the abstract syntax tree, or AST. Each node in the tree represents a syntactic construct in the program, such as an expression, a statement, a function definition, or a class. The shape of the tree captures the nesting and precedence of the original code without preserving the surface details like exact whitespace or parenthesization. Parsing relies on a formal grammar that describes the syntax of the language, and many compilers use generated parsers built from that grammar specification. Some languages allow ambiguous grammars and require the parser to make context dependent decisions, which adds complexity but lets the language feel more natural to write.

After the AST is built, the compiler performs semantic analysis. This is where the rules that go beyond pure syntax are checked. Things like name resolution, type checking, scope analysis, and overload resolution all happen here. The compiler walks over the tree, attaches type information to expressions, links each use of a variable to its declaration, and reports any rule violations as errors. Languages with rich type systems do a lot of work in this phase. Function templates, generics, lifetime constraints, and trait resolution all live here, and a sophisticated compiler will spend a measurable fraction of its total runtime in semantic analysis on real world code.

Once the program is verified to be syntactically and semantically valid, the compiler transforms it into an intermediate representation, often abbreviated to IR. The IR is closer to machine code than the AST, but it is still target independent. Most compilers use a form similar to three address code, where each operation has a destination and at most two source operands. Working in IR makes optimization easier because the language specific clutter has been stripped away and the program is now expressed in terms of simple, well defined operations like load, store, add, multiply, branch, and call. A common choice is static single assignment form, where every variable is assigned exactly once. SSA form makes data flow explicit and unlocks a wide range of optimizations.

Optimization is where compilers earn their reputation. The IR can be transformed in many ways without changing the observable behavior of the program. Constant folding evaluates expressions at compile time when their inputs are known. Dead code elimination removes computations whose results are never used. Common subexpression elimination combines redundant computations into a single one. Loop invariant code motion lifts expressions out of loops if their value does not depend on the loop counter. Inlining replaces a function call with the body of the called function, often unlocking further optimizations because the inliner can now see across the call boundary. Each of these transformations might shave a few percent off the runtime, but together they can make a serious difference, often two to five times faster than the unoptimized code.

The final stage is code generation. Here the IR is mapped down to the actual instruction set of the target machine. The compiler chooses which instructions to emit, how to allocate the limited number of physical registers among the many virtual values in the IR, and how to schedule the instructions to keep the processor pipeline busy. Register allocation is one of the harder problems in this phase, because it is essentially graph coloring with weighted edges, and there are usually more live values at any point than there are physical registers. Instruction scheduling tries to interleave operations in ways that hide memory latency and avoid pipeline stalls. Modern processors are deeply out of order, so the scheduler does not need to be perfect, but a good schedule still helps.

The output of code generation is a sequence of machine instructions encoded as bytes, packaged with metadata describing where to load the code, what symbols it defines, and what symbols it expects to find at link time. The linker then takes one or more of these object files, resolves cross references between them, lays them out in memory according to the target platform's conventions, and produces a final executable. On many systems the linker also handles dynamic libraries, leaving placeholders that the loader fills in when the program actually starts. From the perspective of the original source code this is invisible, but from the perspective of someone debugging a program at runtime it can matter a great deal.

Modern compilers add several more layers on top of this classic pipeline. Just in time compilation defers code generation until the program is already running, so the compiler can specialize for the actual values it sees rather than for every value the type system allows. Profile guided optimization runs the program once with instrumentation, collects information about which branches are taken and which functions are hot, and uses that data to make better decisions on a second compile. Link time optimization waits until every translation unit has been seen, then performs whole program transformations like inlining functions across file boundaries and eliminating dead code that no caller can reach. Each of these techniques trades extra compile time for faster generated code, and modern build systems make it easy to opt in selectively when the trade off is worth it.

Code generation also has to think about the cost of branches. A modern processor speculatively executes instructions ahead of a conditional branch, predicting which way the branch will go and starting work on that path before the actual condition is computed. When the prediction is right, the processor pays nothing for the branch. When it is wrong, the speculative work has to be discarded and the real path started, which costs a dozen or more cycles. Compilers help the predictor by laying out frequently taken paths along the fall through edge, hoisting loop invariants out of branches, and converting some conditionals into branchless arithmetic when the body is small. The same basic ideas apply to indirect calls and function pointers, where the compiler may insert specializations or inline guards so that the common target can be reached without a real branch.

Memory access patterns are another front where the compiler matters more than people often expect. Cache lines are typically sixty four bytes, and a load that misses all the way to main memory can cost two hundred cycles or more. Compilers reorder memory accesses to keep them sequential where possible, hoist loads out of inner loops when the addresses do not change, and emit prefetch hints when a stream of accesses is predictable. Vectorization plays into this too, because a loop that processes one element at a time has very different traffic than the same loop processing eight or sixteen elements per iteration with SIMD instructions. A compiler that can prove the loop is safe to vectorize can collapse what looked like a long sequence of loads, multiplies, and stores into a much smaller stream of wide instructions, often three or four times faster on the same hardware.
